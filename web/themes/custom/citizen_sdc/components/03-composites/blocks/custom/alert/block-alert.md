# Block Alert SDC

The alert block is generated by the `citizen_custom` module, not by a block
content type or a view. This SDC renders only the **empty placeholder**
`<div class="alert-placeholder" data-alert-placeholder="/ajax/alert"></div>`.
The actual alert teasers are injected into that placeholder by
`citizen_custom/alert` JS after fetching `/ajax/alert` on page load.

The companion teaser SDC is `citizen_sdc:node-alert-teaser` — see its
`node-alert-teaser.md` for the other side of the wiring.

## Why AJAX-filled instead of a normal block render

Alerts need to appear and disappear **instantly** when an editor publishes /
unpublishes an alert node or when the current time crosses into or out of the
node's Display Range. A normal block rendering active alerts would be cached
inside the page cache / dynamic page cache / render cache. Invalidating those
caches at display-range edges would require either cron (too slow) or
entity-save hooks (miss the time-based edges entirely).

Splitting it this way keeps the block itself fully page-cacheable (it never
changes), while all per-minute freshness lives in the AJAX endpoint, which is
explicitly uncacheable.

## Module pieces

- **Block plugin:** `web/modules/custom/citizen_custom/src/Plugin/Block/Alert.php`
  (plugin id `alert`, admin label "Alert"). Returns a minimal render array and
  sets `getCacheMaxAge()` to `Cache::PERMANENT` — the placeholder HTML never
  varies per page or per user.
- **AJAX controller:** `web/modules/custom/citizen_custom/src/Controller/Alert.php`
  (route `citizen_custom.alert_ajax`, path `/ajax/alert`). Entity-queries
  published alert nodes whose `field_display_range.value / .end_value` span
  the current unix timestamp, renders them as `teaser` view-mode, and returns
  the concatenated HTML as a raw `Symfony\Component\HttpFoundation\Response`
  with `Cache-Control: no-store, no-cache, must-revalidate, max-age=0` and
  `X-Robots-Tag: noindex`.
- **Route:** `web/modules/custom/citizen_custom/citizen_custom.routing.yml`.
  Declares `options.no_cache: TRUE` to bypass Drupal's page cache on the
  endpoint.
- **Library:** `citizen_custom/alert` in
  `web/modules/custom/citizen_custom/citizen_custom.libraries.yml`. Pulls in
  `web/modules/custom/citizen_custom/js/alert.js`.

## Placement

- **Block placement config:** `config/sync/block.block.citizen_sdc_alert.yml`.
  Placed in the theme's `alert` region with the block label hidden. Visibility
  condition `entity_bundle:node` is negated on the `alert` bundle so the block
  doesn't appear on alert node pages (prevents an alert from rendering inside
  itself).
- **Block template override:** `templates/blocks/alert/block--alert.html.twig`.
  Extends `block.html.twig` and delegates `blockContent` to this SDC — exactly
  the content-only block pattern used by `block-search`.

## Library attachment

`libraryOverrides.dependencies` in `block-alert.component.yml` pulls in two
libraries whenever this SDC renders:

- `citizen_custom/alert` — the fetcher JS (has to be on every page so the
  placeholder gets populated).
- `core/components.citizen_sdc--node-alert-teaser` — the teaser's auto-CSS.
  **This has to be attached here on the parent page.** Any library attached
  during the AJAX render dies with the Response; it never makes it back to
  the page's `<head>`. Attaching on `block-alert` ensures the teaser CSS is
  present before the fetch resolves.

## JS behavior

`js/alert.js` registers `Drupal.behaviors.citizenAlert`:

- On page load, `fetch('/ajax/alert')` → inject HTML into the placeholder →
  call `Drupal.attachBehaviors(placeholder)` so the close-button handlers
  bind to the newly-injected teasers.
- Before injecting, filters out any teaser whose `data-alert-uuid` is in
  `localStorage['citizen_alert_dismissed']` and still within its 30-day
  expiry window.
- Close-button click on a teaser → write `{uuid: Date.now() + 30d}` into the
  dismissal store → remove the teaser from the DOM.
